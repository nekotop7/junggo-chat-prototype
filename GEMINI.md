  
# 프로젝트 안티그래비티 (Anti-Gravity) 전용 지침서 (GEMINI.md)  
  
이 문서는 AI와 협업할 때 반드시 준수해야 하는 규칙입니다. 모든 작업은 이 원칙을 최우선으로 합니다.  
  
## 1. 운영 및 배포 정책  
- **배포 승인:** GitHub 리포지토리 푸시 및 Vercel 배포는 사용자의 명시적인 승인(허락)이 있을 때만 진행할 것.  
- **로컬 우선 원칙:** 모든 기능은 로컬 서버에서 테스트가 완벽히 통과된 후, 결과가 확인된 상태에서만 다음 단계로 넘어가거나 배포를 제안할 것.  
- **상태 보고:** 테스트 결과를 보고할 때는 [상태(성공/실패/주의)], [주요 변경 사항], [다음 단계] 순서로 간결하게 요약할 것.  
  
## 2. 디자인 및 인터랙션 원칙  
- **디자인 준수:** 디자인 시스템의 원칙을 철저히 준수할 것.   
- **임의 변경 금지:** 사용자가 구체적으로 지시하지 않은 디자인적 요소를 AI가 임의로 수정하거나 개선하지 말 것. 변경이 필요하다면 반드시 먼저 제안하고 승인을 받을 것.  
- **인터랙션 표준:** 별도의 디자인 시스템 가이드가 없다면 구글 머테리얼 디자인(Material Design)의 인터랙션 방식을 따를 것.  
  
## 3. 코드 작성 및 개발 규칙  
- **보안 및 환경 설정:** API 키, DB 접속 정보 등 민감한 데이터는 절대 코드에 하드코딩하지 말 것. 반드시 환경 변수(`.env`)를 사용하도록 구조를 잡을 것.  
- **코드 스타일:** [선호하는 스타일: 예 - 함수형 프로그래밍 / 간결한 주석 / 모듈화된 구조]를 준수할 것.  
- **의존성 관리:** 라이브러리 추가는 최소화할 것. 꼭 필요한 경우에만 이유를 설명하고 사용자와 상의한 후 도입할 것.  
- **에러 핸들링:** 예상되는 에러 상황을 미리 정의하고, 에러 발생 시 사용자에게 명확한 원인과 해결 방법을 제안할 것.  
  
## 4. 커뮤니케이션 스타일  
- **명확성:** 사용자의 요구사항이 모호할 경우, 추측해서 작업하지 말고 반드시 추가 질문을 통해 범위를 좁힐 것.  
- **최적화 중심:** 성능과 유지보수를 고려하여 최신 베스트 프랙티스를 따를 것.  
- **간결함:** 불필요한 서론은 줄이고, 작업 내용과 결과 중심의 답변을 할 것.  
  
## 5. 변경 이력  
- 본 문서의 규칙을 수정해야 할 상황이 생기면, AI는 사용자에게 수정 제안을 먼저 할 것.
